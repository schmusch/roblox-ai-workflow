# Blueprint: Gods & Icons — Technical Architecture & Engine Blueprint

Dieses Dokument definiert den vollständigen technischen Architektur- und Implementierungsplan für das Roblox-Erlebnis **„Gods & Icons: Tactical Tycoon & Creed“**. Es transformiert den Game Brief und die Phase-1-Spezifikationen in eine roblox-native Entwicklungs- und Code-Architektur.

---

## 🗺️ 1. Player-Facing Behavior

### Zitadelle (Tycoon & Tempelwirtschaft)
Spieler platzieren brutalistische Altäre und Andachtsstätten auf einem zellularen Grid in ihrer Zitadelle. Diese Strukturen erzeugen kontinuierlich Glauben (Ressource: `Belief`), basierend auf ihrer Andachtsstufe und ihrer Platzierung. 
Das Platzieren wird in Echtzeit lokal predicted, muss aber serverseitig verifiziert werden. Im Abstand von 120 Sekunden können unvorhergesehene Krisen (z. B. *Metaphysische Risse* für Alte Götter, *Daten-Glitch-Vektoren* für Neue Götter) auftreten, welche die Glaubensgenerierung dämpfen und vom Spieler behoben werden müssen.

### Glaubenskrieg (3x3 Grid-Kampf)
Die formlosen Götter treten in einem taktischen, rundenbasierten bzw. asynchronen Auto-Battler auf einem 3x3 Grid an. Der Spieler kann durch göttliche Interventionen aktiv in den Kampf eingreifen. Um eine unmittelbare Reaktion (0ms Input-Verzögerung) zu spüren, rendert der Client Auren und Animationen sofort lokal vor (**Visual Prediction**), während der Server die absolute Autorität behält und über einen 1,5s dauernden **Ascension-Lock** Latenzen maskiert und dem gegnerischen Spieler Zeit für Gegenreaktionen (Counterplay) gibt.

## 🛠️ 2. Roblox Runtime Surfaces Involved

* **Engine Services:**
  * `ReplicatedStorage`: Zur Bereitstellung von Shared-Modulen, Typen, Konfigurationen und RemoteEvents.
  * `ServerScriptService`: Für die Ausführung der unzugänglichen, server-authoritativen Kern-Logik.
  * `DataStoreService` / `ProfileService`: Zur persistenten Speicherung des Zitadellen-Layouts und der Götter-Stammbäume.
  * `RunService`: Zur Synchronisation des 60-Hz-Kampf-Heartbeats.
  * `Players`: Zur Überwachung von Sitzungszeiten und Benutzer-IDs.
* **Instance Classes:**
  * `RemoteEvent`: 4 diskrete Netzwerkkanäle zur Replikation und Validierung.
  * `ModuleScript`: Strukturierte Kapselung von logischen Teildomänen.
  * `Folder`: Logische Strukturierung im `Workspace` (z. B. Grid-Zellen, Götter-Meshes).
  * `Buffer`: Native Luau-`buffer`-Klasse für extrem performante, binäre Speicheroperationen im Kampf.

## ⚔️ 3. Server / Client Ownership Matrix

```
┌────────────────────────────────────────────────────────────────────────┐
│                              REPLICATION                               │
├───────────────────────────────┬────────────────────────────────────────┤
│ SERVER (Authoritative State)   │ CLIENT (Input, Render & Prediction)    │
├───────────────────────────────┼────────────────────────────────────────┤
│ • PlacementModule             │ • PlacementClientController            │
│ • GridCollisionModule         │ • PredictionModule (Aura Highlights)   │
│ • SessionVerificationModule   │ • SLIMController (LOD Mesh Swapping)   │
│ • RollbackModule              │                                        │
└───────────────────────────────┴────────────────────────────────────────┘
```

### Server (ServerScriptService)
* **`PlacementModule`**: Autoritative Verwaltung der Platzierungen, Preisabgleich mit dem serverseitigen Katalog, Fusionsvalidierung.
* **`GridCollisionModule`**: Geometrische Kollisionsprüfung im 3D-Raum (Vermeidung von Mesh-Stacking/Überlappungen).
* **`SessionVerificationModule`**: Absicherung gegen Speedhacks durch server-authoritativen Epoch-Abgleich (`os.time()`).
* **`RollbackModule`**: Historischer Zustandsabgleich im Kampf über einen 90-Frame-Ringpuffer.

### Client (StarterPlayerScripts / PlayerGui)
* **`PlacementClientController`**: Lokale Platzierungsvorschau, Rasterausrichtung, User-Input-Erfassung.
* **`PredictionModule`**: Lokale Effekte bei göttlicher Intervention, Animation des 1,5s Ascension-Locks.
* **`SLIMController`**: Clientseitiges Harmony-System für dynamisches LOD-Rendering zur Performance-Stabilisierung auf Mobilgeräten.

### Shared (ReplicatedStorage)
* **`RingBufferModule`**: Binäre Packing- und Unpacking-Methoden zur bitgenauen Puffer-Modifikation.
* **`DeityCatalog`**: Statische Typdefinitionen, mathematische Radien-Skalierungen ($R_L = R_0 \cdot (1 + \ln(L + 1))$) und Rotationsgeschwindigkeiten ($\omega_L = \frac{\omega_0}{\sqrt{L + 1}}$).

## 💾 5. Data Shape (Luau Typed)

### Zitadellen-Platzierung (`DeityCatalog.lua`)
```luau
export type Faction = "AlteGoetter" | "NeueGoetter"
export type Element = "Erde" | "Daten" | "Feuer" | "Medien" | "Aether" | "Finanzen"

export type CombatRole = "Tank" | "Supporter" | "Slayer" | "Controller"

export type DeityConfig = {
    Id: string,
    Name: string,
    Faction: Faction,
    Element: Element,
    Role: CombatRole,
    BasePrice: number,
    BaseProduction: number, -- Glaubensgenerierung pro Sekunde
    BaseRadius: number,     -- R_0 für orbitalen Halo
    BaseOmega: number,      -- w_0 für Rotationsgeschwindigkeit
}

export type PlacementData = {
    DeityId: string,
    GridCoords: Vector3,
    Rotation: number, -- 0, 90, 180, 270 Grad
    DevotionLevel: number,
}
```

### Binäres Kampf-Speicherlayout (`RingBufferModule.lua`)
Um Luau-Tabellenallokationen bei 60 Hz vollständig zu vermeiden, wird ein pre-allocated `buffer` von 25.560 Bytes verwendet. Dies reicht für eine Historie von 90 Frames (1,5 Sekunden Kampfhistorie bei 60 Hz).

* **Frame-Struktur (Größe: 284 Bytes pro Frame):**
  * `0-3`: Tick-Index (unsigned 32-Bit Integer) $\rightarrow$ 4 Bytes
  * `4-7`: Server-Zeitstempel (double/f64) $\rightarrow$ 8 Bytes
  * `8-277`: 18 Götter-Entitäten (je 15 Bytes pro Entität) $\rightarrow$ 270 Bytes
  * `278-283`: Reserve/Padding $\rightarrow$ 2 Bytes

* **Entitäts-Bitmaske (15 Bytes pro Götter-Entität):**
  * Byte `0`: Active Flag (boolean) $\rightarrow$ 1 Byte
  * Byte `1`: Catalog Index (u8) $\rightarrow$ 1 Byte
  * Byte `2-5`: Owner UserId (u32) $\rightarrow$ 4 Bytes
  * Byte `6-7`: Tile Index (u16) $\rightarrow$ 2 Bytes
  * Byte `8-9`: Health (u16) $\rightarrow$ 2 Bytes
  * Byte `10-11`: Action Points (u16) $\rightarrow$ 2 Bytes
  * Byte `12-14`: Status-Effekt-Bitmaske (24-bit u24) $\rightarrow$ 3 Bytes

## 📡 6. Remote Contracts

### 1. `RequestPlacementEvent` (Client $\rightarrow$ Server)
* **Payload:** `(deityId: string, gridCoords: Vector3, rotation: number)`
* **Server-Validierung:**
  * Typen- und Wertebereichs-Checks (`typeof` Guards).
  * Rate-Limiting: Maximal 4 Platzierungs-Events pro Client und Sekunde.
  * Geometrische Kollisionsprüfung via `GridCollisionModule` (keine überlappenden Meshes / Voxel-Stacking).
  * Besitzprüfung der Entität in der Player-Inventory-Datenbank.
  * Preisprüfung: Abzug des im Server-Katalog hinterlegten Preises (Client-Preissignale werden ignoriert).

### 2. `CancelPlacementEvent` (Client $\rightarrow$ Server)
* **Payload:** `(gridCoords: Vector3)`
* **Server-Validierung:**
  * Existenzprüfung der Entität auf den Koordinaten.
  * Eigentümer-Check (nur der Erbauer darf entfernen).

### 3. `TriggerDivineInterventionEvent` (Client $\rightarrow$ Server)
* **Payload:** `(interventionId: string, targetEntityIndex: number, clientTick: number)`
* **Server-Validierung:**
  * Validierung des Timestamps ($\Delta T = T_{\text{current}} - T_{\text{clientTick}} < 90$).
  * Zustands-Rollback via `RollbackModule` im in-memory Binärpuffer zur exakten Positions- und Statushistorie.
  * Plausibilitätsprüfung des euklidischen Abstands zum Ziel.
  * Ressourcenabzug auf dem Server.

### 4. `SyncCombatBufferEvent` (Server -> Client)
* **Payload:** `(frameBuffer: buffer, lastAuthoritativeTick: number)`
* **Verhalten:** Der Server streamt den serialisierten Kampfzustand mit einer reduzierten Frequenz von **20Hz** (alle 3 Ticks) an alle beteiligten Clients, um die Netzwerk-Pipeline zu schonen.
* **Client-Interpolation:** Der Client interpoliert die Positionsdaten der Götter-Entitäten lokal über die erhaltenen Puffer-Zeitstempel auf 60Hz.

## ⚠️ 7. Failure Modes & Mitigation

### Stale Replication (Latenzverzögerte Client-Platzierung)
* **Szenario:** Ein Client platziert eine Struktur auf einem Voxel, der zeitgleich durch eine Katastrophe blockiert wurde.
* **Mitigation:** Der Server lehnt die Remote-Platzierungsanforderung ab und sendet ein Korrektur-Event mit dem aktuellen Grid-Status an den Client. Der Client korrigiert die lokale Vorschau nahtlos.

### Speedhacking & Time-Dilation
* **Szenario:** Ein exploited Client manipuliert seine lokale Spielgeschwindigkeit (`dt`), um Andachtsressourcen schneller zu generieren.
* **Mitigation:** Die Tempelwirtschaft läuft zu 100 % server-authoritativ. Offline- und Online-Zuwächse werden serverseitig über absolute `os.time()` Zeitstempel berechnet. Client-Zeitmeldungen werden vollständig ignoriert.

### Binärer Buffer-Überlauf (Out-of-Bounds im Ringpuffer)
* **Szenario:** Ein verspäteter Client-Aufruf fordert einen Rollback für einen Tick an, der älter als 90 Frames ($>1,5$ Sekunden) ist.
* **Mitigation:** Das `RollbackModule` verwirft Anforderungen, die die maximale Puffer-Latenz überschreiten. Die Intervention wird serverseitig storniert und der Client wird zur erneuten Synchronisation gezwungen (Interventions-Verwurf).

### DataStore-Verbindungsabbruch während der Sakralen Fusion
* **Szenario:** Ein Spieler fusioniert zwei Götter, die Verbindung bricht ab, bevor die Transaktion in die Roblox-Datenbank geschrieben werden kann.
* **Mitigation:** Einsatz des transaktionalen Profil-Datenbankdesigns (z.B. ProfileService). In-Memory-Warteschlangen sichern Transaktionen. Schlägt der Schreibvorgang nach 3 automatischen Retries fehl, wird die Fusion rückgängig gemacht (Rollback der verbrauchten Entitäten) und das Inventar bleibt konsistent.

## 🧪 8. Verification Path

### Automated Unit- & TDD-Testing (TestEZ)
* **`PlacementModule.spec.lua`**:
  * Simuliert Client-RPC-Spoofing durch Übergabe ungültiger Preismuster.
  * Überprüft, ob das Voxel-Stacking-Schutzsystem überlappende Platzierungen erfolgreich zurückweist.
  * Verifiziert die Einhaltung des Rate-Limiters bei 10 schnellen RPC-Feuerungen.
* **`RingBufferModule.spec.lua`**:
  * Misst das Speicherwachstum im Luau-Heap via `gcinfo()`. Verifiziert, dass ein 1-minütiger simulierter Kampf-Heartbeat eine Speicher-Allokation von exakt $0 \text{ KB}$ aufweist.
  * Überprüft das bitgenaue Packen und Entpacken von Randbereichen (z.B. maximale UserId und maximale Gesundheit).

### Manual & Visual Verification
* **Voxel-Collision-Check:** Manuelle Platzierung von zwei abstrakten Monumenten auf exakt derselben Koordinate im Studio *Play Solo*-Modus. Verifikation, dass der Server die Platzierung ablehnt und der Client visuell zurückgesetzt wird.
* **LOD-Mesh-Swapping-Visualisierung:** Manuelle Navigation im Workspace zur Heavyweight-HL-LL-Grenze. Verifikation über die Render-Ausgabe, dass Götter-Modelle jenseits von 150 Studs nahtlos durch extrem komprimierte SLIM-Meshes ersetzt werden.
