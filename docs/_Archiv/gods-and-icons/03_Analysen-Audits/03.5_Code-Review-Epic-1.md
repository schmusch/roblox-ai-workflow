## 🔒 1. Server-Authority Audit (Anti-Cheat-Schutz)

Das System implementiert eine unüberwindbare server-authoritative Barriere am Netzwerkeingang (`PlacementRemote`):

1. **Typ-Verwirrung (Type Confusion):** In `PlacementModule.HandlePlacementRequest` werden alle vom Client gesendeten Parameter (`deityId`, `gridCoords`, `rotation`) strikt auf ihren Typ geprüft:
   ```luau
   if typeof(deityId) ~= "string" then return end
   if typeof(gridCoords) ~= "Vector3" then return end
   if typeof(rotation) ~= "number" then return end
   ```
   Dies verhindert Exploits, bei denen manipulierte Tabellen oder ungültige Typen gesendet werden, um den Server zum Absturz zu bringen oder Speicherbereiche zu überschreiben.

2. **Spam-Schutz (Rate Limiting):** Ein robuster Zeitstempel-Vergleicher blockiert RPC-Aufrufe desselben Spielers, wenn diese in einem Abstand von weniger als 0,5 Sekunden eintreffen.
   - *Detail:* Der Zeitstempel wird erst *nach* einer erfolgreichen Transaktion aktualisiert, um Spamming mit fehlgeschlagenen Anfragen zu drosseln.

3. **Preis-Verifikation:** Die Monumenten-Preise werden ausschließlich aus dem serverseitig verwalteten Preiskatalog `DEITY_PRICES` geladen (`GridModule.luau`). Client-Signale bezüglich Baukosten werden vollständig ignoriert.

4. **Geometrische Voxel-Validierung (`GridCollisionModule.verifyPlacement`):**
   - **Grenzen-Check (Out of Bounds):** Jede Platzierung prüft geometrisch, ob das Monument über die Grenzen des 10x10 Gitters hinausragt.
   - **Rotations-Kompensation:** Bei einer Drehung um 90° oder 270° werden die Dimensionen (Breite und Tiefe) korrekt vertauscht ($W' \leftarrow D$, $D' \leftarrow W$), bevor die Grenzberechnung stattfindet.
   - **Kollisions-Check:** Jede einzelne betroffene Gitterzelle wird auf Belegung geprüft (`occupiedCells[x][z] ~= nil`). Das Stapeln von Monumenten (Voxel-Stacking) ist serverseitig physikalisch unmöglich.

5. **Transaktionssicherheit:** Der Abzug der Währung (`Belief`) und der Verbrauch des Inventargegenstands werden in `PlayerDataStore` atomar *nach* erfolgreicher Kollisionsprüfung und *vor* dem Workspace-Spawning durchgeführt.

## 🧹 2. Anti-Slop-Prüfung (Code-Reinheit)

Die Implementierung glänzt durch die strikte Vermeidung von aufgeblähtem Enterprise-Jargon:
- Keine pseudo-architektonischen Klassennamen wie `DeityPlacementRepository`, `GridPlacementDTO` oder `VoxelGridAbstractFactory`.
- Strikte Nutzung roblox-konformer Terminologie per `references/roblox-vocabulary.md`:
  - **ModuleScripts** auf dem Server agieren als logische Domänen (`GridModule`, `PlayerDataStore`).
  - **Clientseitige Steuerungen** sind präzise als `Controller` benannt (`PlacementController.client.luau`), was der Roblox-üblichen Client-Architektur entspricht.
- Klare Separation of Concerns: Die Snapping-Berechnung liegt in `GridMath` (Shared), geometrische Integrität in `GridCollisionModule` (Server) und die Darstellung im Controller (Client).

## 🔄 3. Replikations-Audit (Single Source of Truth)

- **Datenfluss (SSOT):** Der Client unterhält keine eigene Schattenkopie des Gitters oder des Inventars. Stattdessen serialisiert der Server den Gitterzustand und das Inventar in JSON-formatierte Attribute auf dem `Player`-Objekt (`CitadelPlacedObjects`, `CitadelInventory`).
- **Ereignisgesteuertes UI:** Clientseitige UI-Komponenten lauschen direkt auf Attributänderungen (`player:GetAttributeChangedSignal`), was Replikationsverzögerungen (Desynchronisation) elegant verhindert.
- **Flimmerfreies Rendering:** Das optische Z-Fighting (Tiefenpuffer-Shimmering) zwischen der Zitadellen-Platte, den Gitterlinien und der Roblox-Baseplate wurde durch saubere mathematische Abstände gelöst:
  - `GridBaseplate` liegt auf `y = 0.1` (Oberkante `y = 0.2`).
  - Gitterlinien liegen auf `y = 0.21`.
  - Standard-Bodenplatte verweilt auf `y = 0.0`.

## 🐛 4. Roblox-spezifische Engine-Härtung

1. **Camera Guardian (Startup Safeguard):**
   Roblox-Kamera-Skripte initialisieren asynchron sehr spät und neigen dazu, clientseitig gesetzte Kameramodi zu überschreiben. Unser **Camera Guardian** umgeht diese Race-Condition meisterhaft:
   - Er bindet sich an das `CharacterAdded`-Event des Spielers.
   - Über ein 3-Sekunden-Fenster prüft und erzwingt er alle 100ms (30 Ticks) den Kameratyp `Custom` und das Kamerasubjekt `Humanoid`.
   - Ein zusätzlicher Listener auf `workspace.CurrentCamera` stellt sicher, dass der Guardian auch bei einem dynamischen Kamerawechsel anspringt.

2. **Ressourcenschonende Ausführung:**
   - Keine veralteten `wait()`- oder `spawn()`-Aufrufe. Die moderne `task`-Bibliothek (`task.wait()`, `task.spawn`) verhindert Thread-Verzögerungen.
   - Automatisches Aufräumen verwaister Spielergitter beim Verlassen des Spiels über `Players.PlayerRemoving`, um Memory Leaks zu verhindern.
