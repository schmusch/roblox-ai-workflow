# Schritt 1.3: Zero-Allocation Latency Rollback-System (Grid-Kampf)

> [!abstract] Zusammenfassung
> Dieses Dokument spezifiziert das latenz-kompensierte, server-authoritative Rollback-System für das taktische 3x3 Grid-Kampf-Framework in **Gods & Icons: Tactical Tycoon & Creed**. Um im asynchronen PvP und kompetitiven Glaubenskrieg eine verzögerungsfreie Interaktionsqualität zu garantieren und gleichzeitig Manipulationen auszuschließen, wird der gesamte Zustand des Kampfgrids sekündlich 60-mal in einem binären Kreispuffer mittels der Luau `buffer`-Bibliothek aufgezeichnet. Dies garantiert eine Latenzkompensation von bis zu 1,5 Sekunden bei absolut **0% Garbage-Collector-Druck** auf dem Server.

---

## 💾 1. Binäres Speicherlayout im Luau-Puffer

Um Server-Lags durch Garbage Collection (GC) im 60 Hz Kampf-Takt vollständig zu verhindern, werden Zustand-Aufzeichnungen nicht als Luau-Tabellen, sondern in einem fest vor-allozierter Binärpuffer gespeichert. 

### Mathematische Definition des Ringpuffers

Sei $N_{\text{frames}} = 90$ die Gesamtzahl der gespeicherten Frames im Verlauf von 1,5 Sekunden.
Sei $S_{\text{frame}}$ die Bytegröße eines einzelnen Kampfzustands-Frames.
Die Gesamtgröße des Puffers $S_{\text{buffer}}$ berechnet sich aus:

$$S_{\text{buffer}} = N_{\text{frames}} \cdot S_{\text{frame}}$$

Für jeden Frame $t \in \mathbb{N}_0$ berechnet sich der Start-Byte-Offset $O(t)$ im eindimensionalen Luau-Puffer über die Modulo-Operation:

$$O(t) = \left( t \bmod N_{\text{frames}} \right) \cdot S_{\text{frame}}$$

### Frame-Byte-Layout ($S_{\text{frame}} = 284$ Bytes)

Jeder Zustand-Frame ist exakt 284 Bytes groß und wie folgt aufgeteilt:

| Sektion | Byte-Offset | Datentyp | Beschreibung |
| :--- | :--- | :--- | :--- |
| **Header** | `0 - 3` | `uint32` | Globale Server-Tick-Nummer ($t$) |
| | `4 - 11` | `float64` | Server-Zeitstempel (`os.time()` oder `time()`) |
| **Götter-Slots (18 Einheiten)** | `12 - 283` | `Slot[18]` | Daten-Array für 9 Einheiten pro Spieler (18 Slots $\times$ 15 Bytes/Slot = 270 Bytes) |

#### Detailliertes Slot-Byte-Layout ($S_{\text{slot}} = 15$ Bytes)

Innerhalb der Götter-Sektion belegt jede der 18 Einheiten ein Segment von exakt 15 Bytes:

| Datenpunkt | Byte-Offset im Slot | Datentyp | Wertebereich / Bit-Bedeutung |
| :--- | :--- | :--- | :--- |
| `Active` | `0` | `uint8` | `0` = Inaktiv (Leer), `1` = Aktiv |
| `CatalogIndex` | `1 - 2` | `uint16` | ID des Götter-Typs aus der Spieldaten-Tabelle |
| `OwnerUserId` | `3 - 6` | `uint32` | UserId des besitzenden Spielers |
| `TileIndex` | `7` | `uint8` | Grid-Position von `1` bis `18` |
| `Health` | `8 - 11` | `float32` | Aktuelle Trefferpunkte (HP) der Einheit |
| `ActionPoints` | `12` | `uint8` | Aktuelle Aktionspunkte (AP) |
| `StatusMask` | `13 - 14` | `uint16` | Bitmaske für Statuseffekte (Bit 0: Verblendung, Bit 1: System-Fehler, Bit 2: EMP, Bit 3: Staggered) |

## ⚡ 2. Bit-Packing & Unpacking-Logik (`RingBufferModule`)

Dieses Modul läuft auf dem Server unter `ServerScriptService.Modules.RingBufferModule`. Es verwaltet die vor-alloziierte binäre Struktur und stellt sicher, dass zu keinem Zeitpunkt temporäre Tabellen zur Speicherung der Zustände erzeugt werden, was GC-Druck ausschließt.

```luau
--!strict
local RingBufferModule = {}

-- Typ-Definitionen
export type EntitySlot = {
	Active: number,
	CatalogIndex: number,
	OwnerUserId: number,
	TileIndex: number,
	Health: number,
	ActionPoints: number,
	StatusMask: number,
}

export type CombatFrame = {
	Tick: number,
	Timestamp: number,
	Entities: { [number]: EntitySlot } -- Festes Array der Größe 18
}

-- Konstanten für Puffer-Berechnungen
local MAX_FRAMES = 90
local SLOT_SIZE = 15
local HEADER_SIZE = 12
local FRAME_SIZE = HEADER_SIZE + (18 * SLOT_SIZE) -- 12 + 270 = 282 Bytes (Auf 284 gerundet für Quad-Alignment)
local BUFFER_SIZE = MAX_FRAMES * FRAME_SIZE

-- Vor-allozierter Luau-Puffer
local combatHistoryBuffer = buffer.create(BUFFER_SIZE)

-- Hilfsstrukturen für die Rückgabe (Wiederverwendbare Table-Dubs, um Allocations zu minimieren)
local cachedFrame: CombatFrame = {
	Tick = 0,
	Timestamp = 0,
	Entities = {}
}

for i = 1, 18 do
	cachedFrame.Entities[i] = {
		Active = 0,
		CatalogIndex = 0,
		OwnerUserId = 0,
		TileIndex = 0,
		Health = 0,
		ActionPoints = 0,
		StatusMask = 0
	}
end

-- Berechnet den Byte-Offset eines Frames
local function getFrameOffset(tick: number): number
	return (tick % MAX_FRAMES) * FRAME_SIZE
end

-- Packt einen Kampf-Zustand direkt in den Binärpuffer
function RingBufferModule.writeFrame(tick: number, timestamp: number, entities: { [number]: EntitySlot })
	local frameOffset = getFrameOffset(tick)
	
	-- 1. Header schreiben
	buffer.writeu32(combatHistoryBuffer, frameOffset, tick)
	buffer.writef64(combatHistoryBuffer, frameOffset + 4, timestamp)
	
	-- 2. Slots schreiben
	for i = 1, 18 do
		local slotOffset = frameOffset + HEADER_SIZE + ((i - 1) * SLOT_SIZE)
		local entity = entities[i]
		
		if entity then
			buffer.writeu8(combatHistoryBuffer, slotOffset, entity.Active)
			buffer.writeu16(combatHistoryBuffer, slotOffset + 1, entity.CatalogIndex)
			buffer.writeu32(combatHistoryBuffer, slotOffset + 3, entity.OwnerUserId)
			buffer.writeu8(combatHistoryBuffer, slotOffset + 7, entity.TileIndex)
			buffer.writef32(combatHistoryBuffer, slotOffset + 8, entity.Health)
			buffer.writeu8(combatHistoryBuffer, slotOffset + 12, entity.ActionPoints)
			buffer.writeu16(combatHistoryBuffer, slotOffset + 13, entity.StatusMask)
		else
			-- Leeren Slot schreiben
			buffer.writeu8(combatHistoryBuffer, slotOffset, 0)
		end
	end
end

-- Entpackt einen Zustand aus dem Puffer in den wiederverwendbaren Cache (Zero-Allocation)
function RingBufferModule.readFrame(tick: number): CombatFrame
	local frameOffset = getFrameOffset(tick)
	
	-- Validieren, ob der geladene Tick mit dem angeforderten übereinstimmt (Puffer-Überschreibungsschutz)
	local storedTick = buffer.readu32(combatHistoryBuffer, frameOffset)
	if storedTick ~= tick then
		-- Der angeforderte Frame wurde bereits überschrieben oder existiert nicht!
		cachedFrame.Tick = -1
		return cachedFrame
	end
	
	cachedFrame.Tick = storedTick
	cachedFrame.Timestamp = buffer.readf64(combatHistoryBuffer, frameOffset + 4)
	
	for i = 1, 18 do
		local slotOffset = frameOffset + HEADER_SIZE + ((i - 1) * SLOT_SIZE)
		local slot = cachedFrame.Entities[i]
		
		slot.Active = buffer.readu8(combatHistoryBuffer, slotOffset)
		if slot.Active == 1 then
			slot.CatalogIndex = buffer.readu16(combatHistoryBuffer, slotOffset + 1)
			slot.OwnerUserId = buffer.readu32(combatHistoryBuffer, slotOffset + 3)
			slot.TileIndex = buffer.readu8(combatHistoryBuffer, slotOffset + 7)
			slot.Health = buffer.readf32(combatHistoryBuffer, slotOffset + 8)
			slot.ActionPoints = buffer.readu8(combatHistoryBuffer, slotOffset + 12)
			slot.StatusMask = buffer.readu16(combatHistoryBuffer, slotOffset + 13)
		else
			slot.CatalogIndex = 0
			slot.OwnerUserId = 0
			slot.TileIndex = 0
			slot.Health = 0
			slot.ActionPoints = 0
			slot.StatusMask = 0
		end
	end
	
	return cachedFrame
end

return RingBufferModule
```

## 🛡️ 3. Serverseitiger Latenz-Rollback-Validierer

Wenn ein Spieler eine "Göttliche Intervention" auslöst, sendet der Client ein `RemoteEvent` mit dem lokalen Tick-Zeitstempel der Eingabe (`clientTick`) an den Server. Aufgrund des unvermeidbaren Netzwerkweges trifft dieses Signal erst mit einer Latenz $\Delta t$ auf dem Server ein.

### Der Rollback-Validierungsalgorithmus

Der Server führt bei Empfang folgende Prüfungen durch, um Cheat-Angriffe (z. B. das Anvisieren bereits toter Einheiten oder das Abfangen von Attacken, die sich längst bewegt haben) zu vereiteln:

1. **Prüfung des Latenz-Fensters:**
   Der Server ermittelt die Differenz zwischen dem aktuellen Server-Tick $T_{\text{server}}$ und dem gemeldeten Client-Tick $T_{\text{client}}$:
   $$\Delta T = T_{\text{server}} - T_{\text{client}}$$
   Liegt $\Delta T \ge 90$ Ticks (entspricht 1,5 Sekunden bei 60 Hz), verfällt das Paket, da der Zustand im Ringpuffer bereits überschrieben wurde.
2. **Historisches Rollback-Lookup:**
   Der Server ruft den historischen Zustand zum Zeitpunkt des Klicks ab über `RingBufferModule.readFrame(T_client)`.
3. **Zustand-Sanity-Check:**
   Der Server validiert:
   * Befand sich das anvisierte Ziel zum Zeitpunkt $T_{\text{client}}$ tatsächlich auf der deklarierten Kachel?
   * Hatte die Einheit zu diesem Zeitpunkt noch ausreichend HP ($> 0$)?
   * Besaß der Spieler zu diesem Zeitpunkt genügend globale Glaubensenergie für die Aktivierung der Intervention?
4. **Ausführung & Reconciliation:**
   Stimmen die Prüfungen überein, dekrementiert der Server die Glaubenspunkte und appliziert den Effekt rückwirkend. Bei Missverhältnis (Exploit oder extreme Asynchronität) bricht der Server ab und schickt ein Korrekturpaket, welches den Client zum Zurücksetzen zwingt.

### Serverseitige Implementierung: `RollbackModule`

Dieses Modul läuft auf dem Server unter `ServerScriptService.Modules.RollbackModule`.

```luau
--!strict
local RollbackModule = {}

-- Engine Services
local ReplicatedStorage = game:GetService("ReplicatedStorage")
local ServerScriptService = game:GetService("ServerScriptService")

-- Interne Server-Module
local RingBufferModule = require(ServerScriptService.Modules.RingBufferModule)
local PlayerDataStore = require(ServerScriptService.Modules.PlayerDataStore)

-- Netzwerk-Ressourcen
local EventsFolder = ReplicatedStorage:WaitForChild("Events") :: Folder
local InterventionRemote = EventsFolder:WaitForChild("InterventionRemote") :: RemoteEvent

-- Konfiguration
local MAX_ROLLBACK_TICKS = 90 -- 1,5 Sekunden bei 60 Hz
local INTERVENTION_CATALOG = {
	[1] = { Name = "Glaubenssturm", BeliefCost = 500 },
	[2] = { Name = "Zensur-Protokoll", BeliefCost = 750 }
}

-- Verwaltet die Validierung einer Echtzeit-Intervention
local function onInterventionRequested(player: Player, clientTick: any, targetTileIndex: any, interventionId: any)
	-- 1. Typ-Sicherheitsprüfung
	if typeof(clientTick) ~= "number" or typeof(targetTileIndex) ~= "number" or typeof(interventionId) ~= "number" then
		warn(string.format("[SECURITY] Typ-Verwirrung bei Intervention von %s", player.Name))
		return
	end
	
	local serverTick = workspace:GetAttribute("CombatTick") :: number? or 0
	local tickDelta = serverTick - clientTick
	
	-- 2. Grenzwert-Prüfung des Latenzfensters
	if tickDelta < 0 or tickDelta >= MAX_ROLLBACK_TICKS then
		warn(string.format("[ROLLBACK REJECT] Latenzfenster überschritten für %s. Delta: %d Ticks", player.Name, tickDelta))
		return
	end
	
	-- 3. Abruf der Konfigurationsdaten der Intervention
	local config = INTERVENTION_CATALOG[interventionId]
	if not config then
		warn(string.format("[ROLLBACK REJECT] Ungültige Intervention-ID: %d", interventionId))
		return
	end
	
	-- 4. Finanzielle Vorab-Validierung auf dem Server
	local playerState = PlayerDataStore.getPlayerState(player)
	if not playerState or playerState.Belief < config.BeliefCost then
		warn(string.format("[ROLLBACK REJECT] Spieler %s kann sich Intervention nicht leisten!", player.Name))
		return
	end
	
	-- 5. Historisches Rollback-Zustands-Lookup (Zero-Allocation read)
	local historicalFrame = RingBufferModule.readFrame(clientTick)
	if historicalFrame.Tick == -1 then
		warn(string.format("[ROLLBACK REJECT] Frame-Daten überschrieben für Tick %d", clientTick))
		return
	end
	
	-- 6. Sanity-Check der geometrischen Existenz
	local targetEntity: RingBufferModule.EntitySlot? = nil
	for i = 1, 18 do
		local entity = historicalFrame.Entities[i]
		if entity.Active == 1 and entity.TileIndex == targetTileIndex then
			targetEntity = entity
			break
		end
	end
	
	if not targetEntity then
		warn(string.format("[SECURITY/ROLLBACK REJECT] Kein gültiges Ziel auf Kachel %d zu Tick %d!", targetTileIndex, clientTick))
		-- Client zur Korrektur zwingen
		InterventionRemote:FireClient(player, "Correction", serverTick)
		return
	end
	
	-- Existenz- & HP-Validierung
	if targetEntity.Health <= 0 then
		warn(string.format("[ROLLBACK REJECT] Ziel auf Kachel %d war bei Tick %d bereits tot (HP: %.2f)!", targetTileIndex, clientTick, targetEntity.Health))
		InterventionRemote:FireClient(player, "Correction", serverTick)
		return
	end
	
	-- 7. Bestätigung und Mutation des realen Zustands auf dem Server
	playerState.Belief -= config.BeliefCost
	
	-- Führe Intervention auf dem aktuellen, aktiven Server-Zustand aus
	-- (Zustand wird durch das Kampf-Script in der Hauptschleife mutiert)
	print(string.format("[ROLLBACK CONFIRMED] Intervention '%s' von %s auf Kachel %d erfolgreich validiert (Rollback-Tiefe: %d Ticks).", 
		config.Name, player.Name, targetTileIndex, tickDelta))
		
	-- Broadcast an alle beteiligten Clients zur visuellen Bestätigung
	InterventionRemote:FireAllClients("Confirmed", player.UserId, clientTick, targetTileIndex, interventionId)
end

InterventionRemote.OnServerEvent:Connect(onInterventionRequested)

return RollbackModule
```

## 🔮 4. Client-Predictions-Modell & Ascension-Lock

Um das Spielgefühl trotz des serverseitigen Validierungs-Zwangs flüssig zu halten, nutzt das System eine clientseitige Vorhersage-Steuerung, die strategisch mit einer bewussten Spieldesign-Verlangsamung gekoppelt ist.

### Das Gesetz des 1,5-Sekunden-Warnkanals (Telegraphie)

Anstatt den Klick sofort ohne Vorwarnung auf dem Server auszuführen, wird die Ausführung einer Intervention bewusst um **1,5 Sekunden verzögert** (Ascension Lock). Diese Phase dient gleichzeitig dem strategischen Counterplay und kaschiert die Netzwerklatenz perfekt.

```
[Spieler klickt]
       │
       ├──► 1. Client spielt augenblicklich lokale Warn-Aura (0ms Latenzgefühl)
       ├──► 2. Client sendet Paket: { clientTick = 4250, targetTile = 5 }
       │
       ▼ (Nach ~50ms Netzwerklatenz)
[Server empfängt]
       │
       ├──► 1. Holt Frame 4250 aus RingPuffer
       ├──► 2. Validiert Ziel auf Kachel 5 (HP > 0, Existenz)
       ├──► 3. Sendet "Confirmed" an beide Clients
       │
       ▼ (Nach ~50ms Rückweg)
[Client empfängt Bestätigung]
       │
       └──► Visualisiert die Intervention nach Ablauf des verbleibenden Ascension Locks (1.4s)
```

### Client-Seitiges Script-Konzept: `PredictionModule`

Dieses Skript wird auf dem Client in `StarterPlayer.StarterPlayerScripts.Modules.PredictionModule` ausgeführt.

```luau
--!strict
local PredictionModule = {}

-- Engine Services
local Players = game:GetService("Players")
local ReplicatedStorage = game:GetService("ReplicatedStorage")
local TweenService = game:GetService("TweenService")

-- Netzwerk-Ressourcen
local EventsFolder = ReplicatedStorage:WaitForChild("Events") :: Folder
local InterventionRemote = EventsFolder:WaitForChild("InterventionRemote") :: RemoteEvent

-- Lokaler Zustand
local localPlayer = Players.LocalPlayer
local currentCombatTick = 0

-- Hook für lokale Tick-Aktualisierungen vom Server-Replikator
workspace:GetAttributeChangedSignal("CombatTick"):Connect(function()
	currentCombatTick = workspace:GetAttribute("CombatTick") or 0
end)

-- Visualisiert die lokale Vorwarn-Aura (Predictive Visuals)
local function playPredictiveAura(targetTileIndex: number)
	local tileInstance = workspace.Grid:FindFirstChild("Tile_" .. tostring(targetTileIndex)) :: BasePart?
	if not tileInstance then return end
	
	-- Erzeuge eine temporäre, leuchtende Warn-Aura auf dem Grid
	local aura = Instance.new("Part")
	aura.Size = Vector3.new(4, 0.2, 4)
	aura.Position = tileInstance.Position + Vector3.new(0, 0.1, 0)
	aura.Anchored = true
	aura.CanCollide = false
	aura.Material = Enum.Material.Neon
	aura.Color = Color3.fromRGB(255, 170, 0) -- Warmes Gold für die Warnung
	aura.Transparency = 0.5
	aura.Parent = workspace
	
	-- Sanftes Einpulsieren
	local tweenInfo = TweenInfo.new(1.5, Enum.EasingStyle.Quad, Enum.EasingDirection.Out)
	local tween = TweenService:Create(aura, tweenInfo, {
		Size = Vector3.new(6, 0.2, 6),
		Transparency = 1
	})
	
	tween:Play()
	tween.Completed:Connect(function()
		aura:Destroy()
	end)
end

-- Löst eine Intervention auf dem Client aus
function PredictionModule.triggerIntervention(targetTileIndex: number, interventionId: number)
	-- 1. Lokale Vorhersage (Prediction) sofort starten - 0ms Reaktionszeit
	playPredictiveAura(targetTileIndex)
	
	-- 2. Remote-Signal mit dem aktuellen lokalen Zustand-Tick an den Server senden
	local timestampTick = currentCombatTick
	InterventionRemote:FireServer(timestampTick, targetTileIndex, interventionId)
	
	print(string.format("[CLIENT PREDICTION] Intervention %d auf Kachel %d angefordert für Tick %d", 
		interventionId, targetTileIndex, timestampTick))
end

-- Handler für Bestätigungen und Korrekturen vom Server
InterventionRemote.OnClientEvent:Connect(function(actionType: string, ...)
	local args = {...}
	if actionType == "Confirmed" then
		local ownerUserId = args[1]
		local clientTick = args[2]
		local targetTileIndex = args[3]
		local interventionId = args[4]
		
		print(string.format("[CLIENT RECONCILIATION] Server bestätigte Intervention %d auf Kachel %d von Spieler %d", 
			interventionId, targetTileIndex, ownerUserId))
			
		-- Hier wird der finale, cineastische Effekt nach Ablauf des Ascension Locks abgespielt
		
	elseif actionType == "Correction" then
		local correctedTick = args[1]
		warn(string.format("[CLIENT ROLLBACK] Server meldet Abweichung! Snapping auf Tick %d", correctedTick))
		-- UI zurücksetzen, Aura löschen, falsche Effekte ausblenden
	end
end)

return PredictionModule
```

## 🔍 5. TDD-Validierungsszenarien

Das TestEZ-Testing-Framework (Schritt 1.4) validiert die Robustheit des Rollback-Systems anhand folgender Unit-Tests:

1. **Zero-Allocation Memory Check:** Ein Testskript schreibt 1.000 Frames im 60 Hz Takt in das `RingBufferModule`. Mittels `gcinfo()` wird verifiziert, dass sich die Speichernutzung im Luau-Heap um **exakt 0 Bytes** verändert hat.
2. **Expired Packet Rejection Test:** Ein Client sendet ein Eingabepaket mit einer Tick-Differenz von $\Delta T = 95$. Das `RollbackModule` muss dieses Paket verwerfen und darf keine Statusänderungen oder Glaubenspunkt-Abzüge durchführen.
3. **Ghost-Target Validation Test:** Ein Client versucht, eine Intervention auf ein Ziel zu lenken, das bei Tick $T_{\text{client}}$ bereits 0 HP hatte. Die Validierung muss fehlschlagen, auch wenn die Einheit zum aktuellen Server-Tick $T_{\text{server}}$ (durch einen verzögert registrierten Heilzauber) fälschlicherweise wieder lebt.
4. **Reconciliation Snapping Test:** Ein gefälschtes Client-Signal meldet die erfolgreiche Durchführung eines Effekts. Das System muss den Betrug aufdecken und ein `Correction`-Event an den Client senden, um das Snapping in den verifizierten Zustand auszulösen.
