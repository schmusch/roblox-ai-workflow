# Roblox Git Master

Git workflows specialised for Roblox repositories. Generic git skills handle commits, branches, merges. Roblox repos have additional artifacts:
- **`wally.lock`** — package lock; should be committed; changes must be intentional.
- **`default.project.json`** — Rojo manifest; conflict-prone; small but important.
- **Place files (`.rbxlx` / `.rbxl`)** — version-controllable only if XML (`.rbxlx`); binary if `.rbxl`. Diffs are huge / unreadable.
- **`sourcemap.json`** — generated; should NOT be committed.
- **Build outputs** — generated `.rbxl` from `rojo build`; should NOT be committed.

This skill captures the conventions and the diagnostics.

## When to use

- Committing or merging Roblox work.
- User asks: "what should be in `.gitignore`", "should I commit the place file", "wally.lock keeps changing".
- Branching strategy for a Roblox team.
- Resolving merge conflicts on Roblox-specific files.
- Setting up a fresh Roblox repo for git.

## When not to use

- Pure runtime / gameplay debugging — wrong skill.
- The user wants to push to a remote and you have no explicit authorization — push back. Per project defaults, do not run `git push` without explicit user request.
- Git operations on a non-Roblox repo — use generic git knowledge, no Roblox-specific overlay needed.

## What this skill produces

Per-task git execution:
- For commits: a sensible commit message + the right set of files staged.
- For merges / rebases: a conflict-resolution plan for Roblox-specific files.
- For setup: a working `.gitignore` and branch conventions.
- Always with a note: **never push to a remote without explicit user authorization**.

## .gitignore baseline for Roblox repos

```gitignore
# Build outputs (generated by Rojo / build scripts)
*.rbxl
!places/baseline.rbxl    # explicit allow if you version a baseline place

# Sourcemaps (generated by Rojo)
sourcemap.json

# Wally
Packages/
DevPackages/
ServerPackages/

# Aftman / Foreman tool installs
.aftman/
.foreman/

# Lune cache
.lune/

# IDE / editor
.vscode/settings.json   # but allow project-shared settings
!.vscode/extensions.json
.idea/

# OS noise
.DS_Store
Thumbs.db
desktop.ini

# Logs / runtime
*.log

```

Adjust per project. **Always commit** `wally.lock`, `aftman.toml` / `foreman.toml`, `default.project.json`, `selene.toml`, `stylua.toml`, `testez.toml`, place files only if `.rbxlx` (text) AND the team has agreed to.

## Place file strategy

This is the most contentious Roblox-git topic. Three viable strategies:

### Strategy A: Place file is canonical (no Rojo, or Rojo as exporter)
- Commit `place.rbxlx` (XML format only — `.rbxl` is binary, near-useless in git).
- Merge conflicts will be brutal; coordinate so only one person edits at a time.
- Treat the place as the source of truth; Rojo (if used) pulls from it.

### Strategy B: Rojo source is canonical (recommended for teams)
- Place file NOT committed; generated by `rojo build` on demand.
- Source of truth lives in `src/` as `.lua` files + small JSON metadata.
- Place artifacts (Models, Decals) are in `assets/` (committed `.rbxm` files or referenced by asset ID).
- Clean diffs; multi-person workflows work.

### Strategy C: Hybrid
- A baseline place (`baseline.rbxlx`) for "world geometry the artist makes in Studio" is committed.
- Code lives in `src/` and syncs via Rojo on top.
- Conflicts only happen on the baseline place; usually rare.

Default recommendation: **Strategy B** unless the project has heavy in-Studio art content.

## Wally lock-file discipline

`wally.lock` should be committed and treated like `package-lock.json` / `Cargo.lock`:
- Every developer gets the same package versions.
- Changes to the lock file should appear in their own commit with an explicit message ("bump Promise@4.0.1 → 4.0.2").
- Do NOT run `wally update` (without args) casually — it updates everything; review the diff before committing.
- If a teammate's lock change conflicts with yours → regenerate (`wally install --check`) on top of the merge base; pick one set of versions.

## Branching conventions (recommended defaults)

- `main` — releasable / shippable.
- `develop` (optional) — integration branch if the team uses Git Flow.
- `feature/<name>` — feature branches, merged via PR.
- `fix/<name>` — bugfix branches.
- `chore/<name>` — toolchain / config / docs changes.

For solo devs: just `main` + `feature/*` is enough. Squash-merge to keep `main` history clean.

## Commit message conventions

Default to **Conventional Commits** style:
- `feat: add party-trading remote validation`
- `fix: dupe path in TradeService when offer rate-limited`
- `chore: bump wally Promise to 4.0.2`
- `docs: clarify server-authority rules in README`
- `refactor: extract InventoryLogic from InventoryService`

For larger changes, include a body explaining **why**, not just what. The diff already shows what.

## Method for typical operations

### Committing a feature
1. Confirm tests / lint / typecheck are green (per `roblox-ultraqa`).
2. `git status` — verify intended files staged, nothing generated leaked in (`sourcemap.json`, `.rbxl`).
3. If `wally.lock` is in the diff but you didn't touch deps → investigate before committing.
4. Write a focused commit message; squash WIPs.
5. **Do not push** to remote without explicit user confirmation.

### Merging a branch
1. Pull latest target branch.
2. Rebase the feature branch on it (or merge — pick per team convention).
3. Resolve conflicts:
   - **Code files** — standard merge; favor the branch with the newer, more complete implementation.
   - **`wally.lock`** — regenerate via `wally install` on the merged manifest.
   - **`default.project.json`** — both sides may have added paths; combine carefully.
   - **`.rbxlx` place file** — if both sides changed, this is painful; pick one and re-apply the other side's intent manually (or merge in Studio side).
4. Re-run all build steps before continuing.

### Rebasing
- Only rebase branches that haven't been shared. Rewriting shared history breaks teammates.
- For interactive rebase: NOT supported in this environment (no interactive flags). Use non-interactive rewrites only, or do interactive rebase locally and report results.

### Resolving a `wally.lock` conflict
1. Discard both lockfile versions.
2. Run `wally install` on the merged `wally.toml`.
3. The new lockfile reflects the resolved manifest.
4. Commit the new lockfile in its own commit.

### Resolving a `default.project.json` conflict
1. JSON conflict markers will appear as text — easy to read.
2. Most likely both sides added a service path; combine them.
3. Run `rojo build` after merge to confirm the manifest is valid.

### Setting up a fresh repo
1. `git init`.
2. Add the `.gitignore` baseline (see above).
3. Initial commit with the manifest files (`wally.toml`, `aftman.toml`, `default.project.json`, `selene.toml`, `stylua.toml`).
4. `wally install` → confirm `wally.lock` generated → commit lock.
5. Set branch protection on remote (when set up) for `main`.

## Anti-pattern checks

- ❌ Committing generated artifacts (`sourcemap.json`, build outputs) — bloat + conflict noise.
- ❌ Committing `Packages/` (Wally output) — that's what the lock file is for.
- ❌ Force-push (`--force` or `--force-with-lease`) without explicit user authorization — disrupts collaborators.
- ❌ `git push` without explicit user request (per workspace defaults).
- ❌ Merge commits in feature branch history — squash-merge to `main` keeps history readable.
- ❌ Committing a place file diff that's 50k lines without summarising what changed gameplay-wise.
- ❌ `wally update` blindly — review the diff.

## Roblox-specific framing

Generic git workflows assume code-only repos. Roblox-specific overlays:
- **Wally `lock` discipline** — like `Cargo.lock` / `package-lock.json` but historically less well-treated.
- **Place file decision** — generic source-control doesn't have a "binary world state" file to worry about.
- **Rojo manifest as schema** — `default.project.json` is structural; mistakes silently break the sync.
- **Aftman / Foreman** — tool-version pinning; lockfile-equivalent for the toolchain itself.

## Handoff

- Clean commit landed → continue work.
- Merge done → re-run `roblox-build-fix` / `roblox-ultraqa` to confirm nothing regressed.
- Wally bump → consider running `roblox-deepsearch` for the changelog of the bumped package.
- About to push to remote → confirm with the user first; never push silently.
