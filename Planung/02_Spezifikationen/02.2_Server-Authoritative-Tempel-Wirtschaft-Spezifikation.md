# Schritt 1.2: Server-Authoritative Validierungs-Architektur (Tempel-Wirtschaft)

> [!abstract] Zusammenfassung
> Dieses Dokument spezifiziert die server-authoritative Validierungs-Architektur für die Tempel-Wirtschaft des Spielprojekts **Gods & Icons: Tactical Tycoon & Creed**. Da Roblox-Clients als standardmäßig kompromittiert betrachtet werden müssen, etabliert diese Architektur eine lückenlose serverseitige Validierung für alle Platzierungs- und Wirtschaftsaktivitäten. Sie schützt das Spiel vor Typ-Verwirrung, Preis-Spoofing, Voxel-Stacking (Kollisions-Exploits) und clientseitiger Zeitmanipulation (Speedhacks).

---

## 🛡️ 1. Das `PlacementRemote` Kommunikations-System

Die Kommunikation zwischen Client und Server bei Platzierungsaktionen erfolgt ausschließlich über ein einziges, unidirektionales `RemoteEvent` namens `PlacementRemote` (gespeichert in `ReplicatedStorage.Events.PlacementRemote`). 

### Datenfluss-Diagramm
```
  [ Client UI / Mouse Input ]
               │
               ▼ (Sendet Anforderung)
    PlacementRemote:FireServer(deityId, gridCoords, orientation)
               │
               ├── [ Server-Script: PlacementModule ]
               │         │
               │         ├──► 1. Rate-Limit & Typ-Validierung (Typ-Sicherheit)
               │         ├──► 2. Eigentums- & Inventarprüfung (Server-State)
               │         ├──► 3. Finanzielle Validierung (Preiskatalog-Abgleich)
               │         └──► 4. Geometrische Kollisionsprüfung (Voxel-Kollision)
               │
               ▼ (Erfolgreich)
  [ Zustand-Mutation im Server-Speicher ] ──► Replikation via Attributes / Remotes ──► Client UI Update
```

### Payload-Schema
Jeder Aufruf an `PlacementRemote` muss exakt die folgenden Parameter enthalten. Jede Abweichung führt zum sofortigen Abbruch der Verarbeitung auf dem Server.

| Parameter | Luau-Typ | Beschreibung | Validierungsschritte auf dem Server |
| :--- | :--- | :--- | :--- |
| `player` | `Player` | Vom Roblox-Engine-Subsystem implizit übergeben. | Darf nicht `nil` sein. Identität wird als absolut sicher angenommen. |
| `deityId` | `string` | Die eindeutige, persistente ID (UUID) der zu platzierenden Gottheit im Inventar des Spielers. | Typ-Prüfung; Existenzprüfung im serverseitigen Spieler-Inventar; Prüfung, ob bereits platziert. |
| `gridCoords` | `Vector3` | Die dreidimensionalen Raster-Koordinaten ($x, y, z$) des Platzierungsursprungs im Tempel. | Typ-Prüfung; Bereichs-Prüfung (Grenzen des aktiven Rasters); Ganzzahligkeitsprüfung (Voxel-Raster). |
| `orientation` | `number` | Die Rotation im Raster in Grad. Gültige Werte: `0`, `90`, `180`, `270`. | Typ-Prüfung; Wertebereichs-Prüfung (`value % 90 == 0` und `0 <= value < 360`). |

### Serverseitige Implementierung: `PlacementModule`

Dieses Modul läuft auf dem Server unter `ServerScriptService.Modules.PlacementModule`. Es implementiert den lückenlosen Schutzschild.

```luau
--!strict
local PlacementModule = {}

-- Engine Services
local Players = game:GetService("Players")
local ReplicatedStorage = game:GetService("ReplicatedStorage")
local ServerScriptService = game:GetService("ServerScriptService")

-- Interne Server-Module
local PlayerDataStore = require(ServerScriptService.Modules.PlayerDataStore)
local GridCollisionModule = require(ServerScriptService.Modules.GridCollisionModule)

-- Netzwerk-Ressourcen
local EventsFolder = ReplicatedStorage:WaitForChild("Events") :: Folder
local PlacementRemote = EventsFolder:WaitForChild("PlacementRemote") :: RemoteEvent

-- Serverseitiger, kanonischer Katalog (Quelle der Wahrheit)
local DEITY_CATALOG = {
	["ancient_stone_01"] = { Name = "Götterfunke der Obsidian-Flut", Footprint = Vector3.new(2, 2, 2), Cost = 1000, Faction = "Alte" },
	["ancient_forest_01"] = { Name = "Zweig der Weltenesche", Footprint = Vector3.new(3, 1, 3), Cost = 2500, Faction = "Alte" },
	["cyber_matrix_01"] = { Name = "Glitch-Kryptochip", Footprint = Vector3.new(1, 2, 1), Cost = 1500, Faction = "Neue" },
	["cyber_server_01"] = { Name = "Neon-Mainframe-Monolith", Footprint = Vector3.new(2, 3, 2), Cost = 5000, Faction = "Neue" }
}

-- Rate-Limiting Konfiguration
local rateLimits = {} :: { [number]: { lastFired: number, count: number } }
local MAX_REQUESTS_PER_SECOND = 4

local function isRateLimited(player: Player): boolean
	local now = os.clock()
	local userId = player.UserId
	
	if not rateLimits[userId] then
		rateLimits[userId] = { lastFired = now, count = 1 }
		return false
	end
	
	local bucket = rateLimits[userId]
	if now - bucket.lastFired >= 1.0 then
		bucket.lastFired = now
		bucket.count = 1
		return false
	end
	
	bucket.count += 1
	if bucket.count > MAX_REQUESTS_PER_SECOND then
		warn(string.format("[SECURITY] Player %s (%d) wurde wegen Placement-Spam blockiert!", player.Name, userId))
		return true
	end
	
	return false
end

-- Bereinigung der Rate-Limits beim Verlassen des Spielers
Players.PlayerRemoving:Connect(function(player)
	rateLimits[player.UserId] = nil
end)

-- Zentraler Handler für die Platzierungsanforderung
local function onPlacementRequested(player: Player, deityId: any, gridCoords: any, orientation: any)
	-- 1. Rate-Limiting Prüfung
	if isRateLimited(player) then return end
	
	-- 2. Typ-Sicherheits-Validierung (Schutz vor Typ-Verwirrung)
	if typeof(deityId) ~= "string" then
		warn(string.format("[SECURITY] Typ-Verwirrung deityId von %s: %s", player.Name, typeof(deityId)))
		return
	end
	if typeof(gridCoords) ~= "Vector3" then
		warn(string.format("[SECURITY] Typ-Verwirrung gridCoords von %s: %s", player.Name, typeof(gridCoords)))
		return
	end
	if typeof(orientation) ~= "number" then
		warn(string.format("[SECURITY] Typ-Verwirrung orientation von %s: %s", player.Name, typeof(orientation)))
		return
	end
	
	-- 3. Wertebereichs-Validierung für Rotation
	local rot = math.floor(orientation + 0.5)
	if rot ~= 0 and rot ~= 90 and rot ~= 180 and rot ~= 270 then
		warn(string.format("[SECURITY] Ungültige Rotation von %s: %d", player.Name, rot))
		return
	end
	
	-- 4. Ganzzahligkeitsprüfung der Koordinaten (Voxel-Raster-Rundung)
	local roundedCoords = Vector3.new(
		math.floor(gridCoords.X + 0.5),
		math.floor(gridCoords.Y + 0.5),
		math.floor(gridCoords.Z + 0.5)
	)
	
	-- 5. Abruf der kanonischen Spielerdaten vom Server-Speicher
	local playerState = PlayerDataStore.getPlayerState(player)
	if not playerState then
		warn(string.format("[ERROR] Kein Spieler-Zustand für %s gefunden!", player.Name))
		return
	end
	
	-- 6. Inventar- & Eigentumsprüfung (Befindet sich die Gottheit im Inventar des Spielers?)
	local inventoryDeity = playerState.Inventory[deityId]
	if not inventoryDeity then
		warn(string.format("[SECURITY] Spieler %s besitzt deityId %s nicht!", player.Name, deityId))
		return
	end
	
	if inventoryDeity.IsPlaced then
		warn(string.format("[SECURITY] Gottheit %s von %s ist bereits platziert!", deityId, player.Name))
		return
	end
	
	-- 7. Katalog-Lookup & Preis-Verifikation
	local itemCatalogTemplate = DEITY_CATALOG[inventoryDeity.CatalogKey]
	if not itemCatalogTemplate then
		warn(string.format("[ERROR] CatalogKey %s existiert nicht!", inventoryDeity.CatalogKey))
		return
	end
	
	-- 8. Finanzielle Validierung (Kann sich der Spieler die Platzierungsgebühr leisten?)
	-- Hinweis: Die Kosten stammen aus dem server-eigenen Katalog, nicht vom Client!
	local placementFee = itemCatalogTemplate.Cost
	if playerState.Belief < placementFee then
		warn(string.format("[SECURITY] Spieler %s kann sich Gebühr %d nicht leisten (Aktuell: %d)", player.Name, placementFee, playerState.Belief))
		return
	end
	
	-- 9. Geometrische Kollisionsprüfung (Voxel-Raster & Stack-Check)
	local isPlacementValid = GridCollisionModule.verifyPlacement(
		playerState.Grid,
		itemCatalogTemplate.Footprint,
		roundedCoords,
		rot
	)
	
	if not isPlacementValid then
		warn(string.format("[SECURITY] Geometrischer Validierungs-Fehlversuch für %s bei %s", player.Name, tostring(roundedCoords)))
		return
	end
	
	-- 10. Atomare Zustands-Mutation auf dem Server
	playerState.Belief -= placementFee
	inventoryDeity.IsPlaced = true
	inventoryDeity.PlacedPosition = roundedCoords
	inventoryDeity.PlacedRotation = rot
	
	-- Zellen als belegt markieren
	GridCollisionModule.markCellsOccupied(
		playerState.Grid,
		deityId,
		itemCatalogTemplate.Footprint,
		roundedCoords,
		rot
	)
	
	-- Replikation an den Client (Server -> Client)
	PlayerDataStore.replicateState(player)
end

-- Event-Anbindung
PlacementRemote.OnServerEvent:Connect(onPlacementRequested)

return PlacementModule
```

## 📐 2. Serverseitige Voxel-Kollisionsprüfung & Grid-Validierung

Um betrügerische Platzierungen zu verhindern – insbesondere das Ineinanderschachteln von Modellen oder die Platzierung außerhalb der freigeschalteten Zonen ("Voxel-Stacking") –, berechnet der Server die exakte geometrische Ausdehnung der Götter-Entitäten in Echtzeit.

### Mathematische Rotations-Formulierung

Sei ein Objekt mit der ursprünglichen Größe $(W, H, D)$ (Breite, Höhe, Tiefe) gegeben. Wird das Objekt um die Hochachse ($Y$-Achse) um den Winkel $\theta \in \{0^\circ, 90^\circ, 180^\circ, 270^\circ\}$ gedreht, verändert sich seine Ausdehnung im horizontalen Koordinatensystem wie folgt:

$$W' = \begin{cases} W & \text{falls } \theta \in \{0^\circ, 180^\circ\} \\ D & \text{falls } \theta \in \{90^\circ, 270^\circ\} \end{cases}$$

$$D' = \begin{cases} D & \text{falls } \theta \in \{0^\circ, 180^\circ\} \\ W & \text{falls } \theta \in \{90^\circ, 270^\circ\} \end{cases}$$

$$H' = H$$

Die Menge der durch das platzierte Objekt eingenommenen Voxel-Koordinaten $F$ ausgehend vom Platzierungspunkt $(X_0, Y_0, Z_0)$ ist definiert als:

$$F(X_0, Y_0, Z_0, \theta) = \left\{ (x, y, z) \in \mathbb{Z}^3 \;\middle|\; \begin{aligned} X_0 \le x < X_0 + W' \\ Y_0 \le y < Y_0 + H' \\ Z_0 \le z < Z_0 + D' \end{aligned} \right\}$$

Eine Platzierungsanforderung ist geometrisch **nur dann gültig**, wenn die beiden folgenden Bedingungen simultan erfüllt sind:

1. **Gittergrenzen-Einhaltung (Boundary Containment):**
   $$F(X_0, Y_0, Z_0, \theta) \subseteq \text{GridBounds}$$
   Wobei $\text{GridBounds}$ das freigeschaltete Gittervolumen des Spielers darstellt:
   $$\text{GridBounds} = \{ (x, y, z) \in \mathbb{Z}^3 \mid 0 \le x < G_{\max, x}, \; 0 \le y < G_{\max, y}, \; 0 \le z < G_{\max, z} \}$$

2. **Kollisionsfreiheit (Overlap Block):**
   $$\forall c \in F(X_0, Y_0, Z_0, \theta): \quad \text{Grid}[c] = \emptyset \quad (\text{Zelle ist leer})$$

### Serverseitige Implementierung: `GridCollisionModule`

Dieses Modul wird in `ServerScriptService.Modules.GridCollisionModule` implementiert und kapselt alle geometrischen Algorithmen im dreidimensionalen Raum-Gitter.

```luau
--!strict
local GridCollisionModule = {}

-- Definition der Gitterstrukturen
export type GridCell = {
	OccupiedByDeityId: string?,
}

export type PlayerGrid = {
	Width: number,
	Height: number,
	Depth: number,
	Cells: { [string]: GridCell } -- Key-Format: "x_y_z"
}

-- Hilfsfunktion: Konvertiert 3D-Koordinaten in den eindeutigen Gitter-Schlüssel
local function getCellKey(x: number, y: number, z: number): string
	return string.format("%d_%d_%d", x, y, z)
end

-- Berechnet die rotierten Dimensionen eines Objekts
function GridCollisionModule.getRotatedDimensions(footprint: Vector3, rotation: number): (number, number, number)
	local w = math.floor(footprint.X + 0.5)
	local h = math.floor(footprint.Y + 0.5)
	local d = math.floor(footprint.Z + 0.5)
	
	if rotation == 90 or rotation == 270 then
		return d, h, w  -- Swap Breite und Tiefe bei 90° und 270°
	end
	return w, h, d
end

-- Validiert, ob eine Platzierung den geometrischen Regeln entspricht
function GridCollisionModule.verifyPlacement(grid: PlayerGrid, footprint: Vector3, position: Vector3, rotation: number): boolean
	local px, py, pz = position.X, position.Y, position.Z
	local w, h, d = GridCollisionModule.getRotatedDimensions(footprint, rotation)
	
	-- 1. Grenzenprüfung (Boundary Check)
	if px < 0 or py < 0 or pz < 0 then
		return false
	end
	if px + w > grid.Width or py + h > grid.Height or pz + d > grid.Depth then
		warn(string.format("[PLACEMENT REJECT] Außerhalb der Grenzen: Position %s, Rotierte Größe (%d,%d,%d), Gitter-Größe (%d,%d,%d)", 
			tostring(position), w, h, d, grid.Width, grid.Height, grid.Depth))
		return false
	end
	
	-- 2. Kollisionsprüfung (Overlap & Stack Check)
	for x = px, px + w - 1 do
		for y = py, py + h - 1 do
			for z = pz, pz + d - 1 do
				local cellKey = getCellKey(x, y, z)
				local cell = grid.Cells[cellKey]
				
				if cell and cell.OccupiedByDeityId ~= nil then
					warn(string.format("[PLACEMENT REJECT] Kollision auf Zelle (%d,%d,%d) mit Gottheit %s", 
						x, y, z, tostring(cell.OccupiedByDeityId)))
					return false -- Voxel-Überlagerung / Stacking blockiert!
				end
			end
		end
	end
	
	return true
end

-- Markiert Voxel im Gitter als belegt
function GridCollisionModule.markCellsOccupied(grid: PlayerGrid, deityId: string, footprint: Vector3, position: Vector3, rotation: number)
	local px, py, pz = position.X, position.Y, position.Z
	local w, h, d = GridCollisionModule.getRotatedDimensions(footprint, rotation)
	
	for x = px, px + w - 1 do
		for y = py, py + h - 1 do
			for z = pz, pz + d - 1 do
				local cellKey = getCellKey(x, y, z)
				grid.Cells[cellKey] = {
					OccupiedByDeityId = deityId
				}
			end
		end
	end
end

-- Gibt belegte Zellen wieder frei (bei Entfernung/Umgruppierung)
function GridCollisionModule.clearCells(grid: PlayerGrid, footprint: Vector3, position: Vector3, rotation: number)
	local px, py, pz = position.X, position.Y, position.Z
	local w, h, d = GridCollisionModule.getRotatedDimensions(footprint, rotation)
	
	for x = px, px + w - 1 do
		for y = py, py + h - 1 do
			for z = pz, pz + d - 1 do
				local cellKey = getCellKey(x, y, z)
				grid.Cells[cellKey] = {
					OccupiedByDeityId = nil
				}
			end
		end
	end
end

return GridCollisionModule
```

## ⏱️ 3. Sitzungsdauer-Verifikation & Schutz vor Offline-AFK-Rendite-Exploits

### Das Bedrohungsszenario (Client-seitige Zeitdilatation)
Roblox-Exploits (wie *Cheat Engine Speedhacks* oder manipulierter Client-Code) können die lokale Engine-Geschwindigkeit (`task.wait` Frequenz) beschleunigen. Verlässt sich der Server auf clientseitig gemeldete Zeitintervalle oder Client-Yields, können Angreifer Stunden der In-Game-Akkumulation in Sekunden simulieren und das Glaubenssystem aushebeln.

### Die Schutzarchitektur (Serverseitige Epoch-Zeitmessung)
Um dies mathematisch auszuschließen, wendet das Framework drei Sicherheits-Ebenen an:
1. **Server-Heartbeat:** Die Glaubensgenerierung wird ausschließlich serverseitig in festen Intervallen über `RunService.Heartbeat` gesteuert. Client-Signale zur Ertrags-Ausschüttung existieren nicht.
2. **Epoch-Zeitmessung:** Sitzungszeiten werden ausschließlich über die Roblox-eigene Implementierung von `os.time()` (Server-seitige Unix-Epoche in Sekunden) gemessen. Diese läuft unabhängig von Client-Frames und Client-Taktungen.
3. **Sichere Offline-Simulation:** Beim Betreten des Spiels wird die akkumulierte Offline-Zeit anhand der Differenz zwischen dem aktuellen serverseitigen `os.time()` und dem in `DataStoreService` persistierten letzten Speicher-Zeitpunkt berechnet.

### Mathematische Formulierung der Offline-Progression und Dämpfung
Um Bot-Grinding unattraktiv zu machen und Spieler zur regelmäßigen Rückkehr zu motivieren, wird die passive Offline-Generierungszeit gedämpft und gedeckelt. 

Sei $t_{\text{now}}$ die aktuelle Serverzeit, $t_{\text{last\_save}}$ der Unix-Zeitstempel des letzten server-bestätigten Speichervorgangs, $R_p$ die kumulierte Glaubensproduktionsrate aller platzierten Götter des Spielers, und $T_{\text{cap}} = 43200$ Sekunden (exakt 12 Stunden) die harte Grenze für ungedämpfte Erträge.

Die mathematische Offline-Zeitdifferenz $\Delta t$ berechnet sich als:

$$\Delta t = t_{\text{now}} - t_{\text{last\_save}}$$

Der tatsächlich anrechenbare, ungedämpfte Zeitraum $t_{\text{effektiv}}$ ist:

$$t_{\text{effektiv}} = \min(\Delta t, T_{\text{cap}})$$

Die überschüssige Offline-Dauer $t_{\text{overflow}}$ beträgt:

$$t_{\text{overflow}} = \max(0, \Delta t - T_{\text{cap}})$$

Um den passiven Ertrag nach der 12-Stunden-Grenze progressiv einzuschränken, greift der Dämpfungsfaktor $\gamma = 0{,}10$ (90% Dämpfung). Die finale Glaubensausschüttung $B_{\text{offline}}$ berechnet sich somit server-seitig über:

$$B_{\text{offline}} = R_p \cdot \left( t_{\text{effektiv}} + \gamma \cdot t_{\text{overflow}} \right)$$

Durch diesen mathematischen Riegel verliert Botting jenseits der 12-Stunden-Marke 90 % seiner Effizienz, während aktive Spieler belohnt werden.

### Serverseitige Implementierung: `SessionVerificationModule`

Dieses Modul läuft unter `ServerScriptService.Modules.SessionVerificationModule` und steuert die Zeithygiene.

```luau
--!strict
local SessionVerificationModule = {}

-- Engine Services
local Players = game:GetService("Players")
local ServerScriptService = game:GetService("ServerScriptService")

-- Interne Server-Module
local PlayerDataStore = require(ServerScriptService.Modules.PlayerDataStore)

-- Konstanten
local OFFLINE_EARNINGS_CAP_SECONDS = 43200 -- 12 Stunden Hard-Cap für 100% Effizienz
local DAMPING_FACTOR_GAMMA = 0.10          -- 90% Dämpfung nach 12 Stunden offline

-- Berechnet die offline akkumulierten Glaubenspunkte sicher auf dem Server
function SessionVerificationModule.processOfflineEarnings(player: Player)
	local playerState = PlayerDataStore.getPlayerState(player)
	if not playerState then return end
	
	local now = os.time()
	local lastSaved = playerState.LastSavedTimestamp
	
	-- Schutz vor Zeitreise-Exploits (Falls Server-Zeit asynchron ist oder Daten manipuliert wurden)
	if lastSaved <= 0 or lastSaved >= now then
		playerState.LastSavedTimestamp = now
		return
	end
	
	local deltaTime = now - lastSaved
	if deltaTime < 10 then return end -- Keine Ausschüttung bei extrem kurzen Fluktuationen
	
	-- Berechnung der effektiven Zeiten gemäß Dämpfungs-Formel
	local effectiveTime = math.min(deltaTime, OFFLINE_EARNINGS_CAP_SECONDS)
	local overflowTime = math.max(0, deltaTime - OFFLINE_EARNINGS_CAP_SECONDS)
	
	-- Ertragsrate berechnen (Glauben pro Sekunde)
	local productionRate = playerState.BeliefRatePerSecond
	
	-- Formel-Anwendung: B_offline = R_p * (t_effektiv + gamma * t_overflow)
	local baseEarnings = productionRate * effectiveTime
	local dampedEarnings = productionRate * DAMPING_FACTOR_GAMMA * overflowTime
	local totalOfflineEarnings = math.floor(baseEarnings + dampedEarnings)
	
	if totalOfflineEarnings > 0 then
		playerState.Belief += totalOfflineEarnings
		playerState.LastSavedTimestamp = now
		
		print(string.format("[OFFLINE PROG] Spieler %s war %d Sekunden offline. Erhaltener Glaube: %d (Gedämpft: %s)", 
			player.Name, deltaTime, totalOfflineEarnings, tostring(overflowTime > 0)))
	end
end

-- Heartbeat-Loop zur periodischen Sicherung der Server-Zeit (Verhindert Verlust bei Server-Crashs)
function SessionVerificationModule.startHeartbeatTimer(player: Player)
	task.spawn(function()
		while player.Parent == Players do
			task.wait(60) -- Alle 60 Sekunden Synchronisation
			local playerState = PlayerDataStore.getPlayerState(player)
			if playerState then
				playerState.LastSavedTimestamp = os.time()
				-- Optional: In-Memory Datensicherung
			end
		end
	end)
end

-- Listener für den Beitritt
Players.PlayerAdded:Connect(function(player)
	-- Initialisierung abwarten, bis Daten geladen sind
	PlayerDataStore.awaitDataLoaded(player)
	
	-- Offline-Erträge verarbeiten
	SessionVerificationModule.processOfflineEarnings(player)
	
	-- Heartbeat-Timer starten
	SessionVerificationModule.startHeartbeatTimer(player)
end)

-- Listener für das Verlassen (Kanonischer Zeitstempel für Offline-Berechnung sichern)
Players.PlayerRemoving:Connect(function(player)
	local playerState = PlayerDataStore.getPlayerState(player)
	if playerState then
		playerState.LastSavedTimestamp = os.time()
		PlayerDataStore.saveAndReleaseData(player)
	end
end)

return SessionVerificationModule
```

## 🔍 4. Testbarkeit & Validierungs-Audit-Checkliste

Zur Überprüfung dieser Architektur im Rahmen des TDD-Frameworks (Schritt 1.4) werden folgende Test-Szenarien mittels **TestEZ** implementiert:

1. **Adversarial Type Spoofing Test:** Client ruft `PlacementRemote` mit manipulierten Typen auf (z. B. `deityId = 1234` oder `gridCoords = "invalid"`). Der Server muss diesen Versuch abfangen und darf den Speicherzustand nicht korrumpieren.
2. **Boundary Collision (Glitch Test):** Eine Gottheit der Größe $2 \times 2 \times 2$ wird an Position $(15, 0, 15)$ in einem $16 \times 16 \times 16$ Gitter angefordert. Da die Drehung die Grenzen überschreiten würde, muss der Server die Anfrage verwerfen.
3. **Overlapping Voxel Test:** Zwei Gottheiten werden an exakt derselben Koordinate angefordert. Die zweite Anforderung muss fehlschlagen und die Belegung der Zellen muss unangetastet bleiben.
4. **Time Dilation Resiliency Test:** Ein simulierter Client meldet einen Zeitfluss von $10\times$ Realgeschwindigkeit. Die serverseitige `os.time()` Überprüfung muss die manipulierten Zeittakte verwerfen und die Produktion stur am Epoch-Takt ausrichten.
